✅ Scenario 1 – Scaling an Ecommerce Platform’s MySQL Database
Business Context

An ecommerce platform experiences a surge in traffic during holiday seasons. Customer orders, product searches, and payment records increase tenfold.

Challenges

Increased read/write load on the database.

Avoiding performance degradation.

Supporting complex reporting without slowing down transactions.

Scaling Strategies

Read Replicas (Horizontal Scaling):

Set up one or more read replicas to handle read-heavy queries like product searches and reports.

Configure application routing logic (via JDBC, load balancer, or application code) to direct reads to replicas.

Commands:

-- Enable binary logging on master for replication:
[mysqld]
log-bin=mysql-bin
server-id=1


On replica:

CHANGE MASTER TO
MASTER_HOST='master-db',
MASTER_USER='replica_user',
MASTER_PASSWORD='password',
MASTER_LOG_FILE='mysql-bin.000001',
MASTER_LOG_POS=154;
START SLAVE;


Sharding (Partitioning Data by Customer or Region):

Split large tables across multiple servers by customer ID ranges or geographical regions.

Example: Customers with IDs 1–1,000,000 in shard A, 1,000,001–2,000,000 in shard B.

Caching with Redis or Memcached:

Store frequent queries or session data in cache layers.

Reduces pressure on the database during peak loads.

Connection Pooling in Java:

Use connection pools (e.g., HikariCP) to manage database connections efficiently.

Controls resource usage and prevents overload.

✅ Scenario 2 – Backups for Compliance and Disaster Recovery
Business Context

A financial services application must ensure data integrity, auditability, and disaster recovery, complying with industry regulations.

Challenges

Avoid data loss during hardware failures.

Ensure backups are consistent and quickly recoverable.

Encrypt sensitive data during storage and transit.

Backup Strategies

Logical Backup Using mysqldump:

Suitable for small-to-medium datasets.

mysqldump -u admin -p --single-transaction --routines --triggers ecommerce_db > ecommerce_backup.sql


--single-transaction ensures consistency for InnoDB tables without locking them.

Physical Backup Using mysqlbackup or Percona XtraBackup:

Fast for large datasets.

Performs incremental backups to reduce overhead.

Point-in-Time Recovery (PITR):

Combine base backup with binary logs to replay transactions up to a desired timestamp.

mysqlbinlog mysql-bin.000001 | mysql -u admin -p ecommerce_db


Automated Backup Scheduling:

Use cron jobs to schedule regular backups.

0 2 * * * /usr/local/bin/backup_script.sh


Encrypted Backups:

Use tools like openssl or cloud provider encryption to secure backup files.

✅ Scenario 3 – Failover for High Availability
Business Context

A payment processing system must remain online 24/7. Even a short downtime can result in lost revenue and customer trust.

Challenges

Avoid single points of failure.

Detect and switch to standby systems automatically.

Prevent split-brain scenarios where multiple masters run at once.

Failover Strategies

Master-Slave Failover Using Orchestration Tools:

Tools like MHA (Master High Availability) or Orchestrator monitor replication health and promote a new master if the primary fails.

Automatic Health Checks:

Implement monitoring using Prometheus or Percona Monitoring and Management (PMM).

Set up alerts if replication lags or if the master is unreachable.

Use Virtual IP (VIP) or Proxy Layer:

The application connects via a floating IP or proxy server (e.g., HAProxy, ProxySQL).

When failover occurs, only the proxy endpoint needs to be updated.

Avoid Split-Brain:

Use quorum-based election or fencing mechanisms to ensure only one master is active.

Regular Failover Drills:

Schedule tests to simulate failures and ensure recovery processes work smoothly.

4) Problematic query types — examples, why they’re bad, and how to fix them

I list the most common classes of “bad” queries you’ll see in interviews and production, with examples + stepwise fixes.

A. Full table scan due to missing index

Example

SELECT * FROM orders WHERE customer_id = 12345;  -- no index on customer_id


Why bad: Scans every row → O(N) cost on large tables.
Detect: EXPLAIN shows type = ALL and huge rows. 
MySQL

Fix (stepwise):

Add index:

CREATE INDEX idx_orders_customer ON orders(customer_id);


Re-run EXPLAIN — should show index or ref (seek).

If queries also filter by date/order, use composite index (customer_id, order_date).

B. Non-SARGable predicates (functions on columns)

Example

SELECT * FROM orders WHERE YEAR(order_date) = 2024;


Why bad: Function on column prevents index usage.
Detect: EXPLAIN shows type = ALL.
Fix: Rewrite as range:

WHERE order_date >= '2024-01-01' AND order_date < '2025-01-01';


Or create a functional/generated column and index it (MySQL 8.x supports functional/generator indexes). 
Medium
+1

C. Leading-wildcard LIKE '%abc' (no index use)

Example

SELECT * FROM products WHERE sku LIKE '%X-123%';


Why bad: Leading wildcard prevents index seek → full scan.
Detect: EXPLAIN with full scan.
Fixes:

Use full-text search / MATCH() / FULLTEXT index for natural language.

Use external search (Elasticsearch) for complex substring search.

If pattern is suffix/prefix only (no leading %), index works.

D. Correlated subquery executed per row (expensive)

Example

SELECT c.*, (SELECT SUM(amount) FROM orders o WHERE o.customer_id = c.id) total
FROM customers c;


Why bad: Subquery runs once per customer → large cost.
Detect: High time; EXPLAIN shows nested loop.
Fix: Precompute and JOIN:

SELECT c.*, COALESCE(o.total,0) total
FROM customers c
LEFT JOIN (
  SELECT customer_id, SUM(amount) AS total
  FROM orders
  GROUP BY customer_id
) o ON o.customer_id = c.id;

E. Cartesian Products (missing join condition)

Example

SELECT * FROM a, b;  -- no join condition


Why bad: Rows multiply → enormous resultset.
Detect: EXPLAIN shows no keys and huge rows.
Fix: Add proper join condition or WHERE clause; use explicit JOIN ... ON ... to avoid accidental carts.

F. ORDER BY causing filesort (no index to satisfy ordering)

Example

SELECT id, name FROM orders WHERE status='PENDING' ORDER BY created_at DESC LIMIT 100;


Why bad: Without index on (status, created_at) MySQL may do filesort and temp tables.
Fix: Add composite index:

CREATE INDEX idx_orders_status_created ON orders(status, created_at);


Then EXPLAIN should show index usage to satisfy ORDER BY + LIMIT.

G. Large IN lists or NOT IN with NULL problems

Example

SELECT * FROM items WHERE id IN (1,2,3,...,5000);


Why bad: Long SQL, parsing overhead; NOT IN with NULLs gives surprising results.
Fix: Put IDs into a temp table (or use JOIN), or pass IDs as a table-valued parameter; for NOT IN, use NOT EXISTS or handle NULLs.

H. Heavy aggregations on raw tables (GROUP BY on millions of rows)

Example

SELECT product_id, SUM(amount) FROM orders GROUP BY product_id;


Why bad: Scans whole table every time; expensive for dashboards.
Fixes:

Maintain incremental summary table or materialized view (pre-aggregate into daily_product_sales and query that).

Add covering index to help GROUP BY if possible.

I. Joins on non-indexed foreign keys

Example

SELECT * FROM orders o JOIN customers c ON o.customer_id = c.id;


If customers.id or orders.customer_id not indexed, join is slow.
Fix: Ensure both sides are indexed (primary key on customers.id and index on orders.customer_id).

J. Long-running transactions (locks, replication lag, deadlocks)

Bad pattern

Begin tx, do multiple updates with user interaction, then commit → locks held long.
Consequences: Locks, deadlocks, replication lag.
Fix: Keep transactions short, avoid user interaction inside tx, batch updates, set proper isolation level and retry on deadlock. For debugging: SHOW ENGINE INNODB STATUS\G to inspect latest deadlock. 
Percona

K. Using SELECT * with BLOB/JSON fields

Problem: Transfers huge data unnecessarily.
Fix: Select only needed columns; store large BLOBs in separate storage or separate table if necessary.

L. Using DISTINCT on large sets

Issue: DISTINCT forces extra work (sort/uniq).
Fix: If duplicates are rare and you can prevent them upstream (unique constraints), avoid DISTINCT. Otherwise use indexing/design changes.

2) Correlated subquery (bad → good)

Bad:

SELECT c.name, (SELECT COUNT(*) FROM orders o WHERE o.customer_id = c.id) cnt
FROM customers c;


Good:

SELECT c.name, COALESCE(o.cnt,0) cnt
FROM customers c
LEFT JOIN (
  SELECT customer_id, COUNT(*) cnt
  FROM orders
  GROUP BY customer_id
) o ON o.customer_id = c.id;

3) ORDER BY + LIMIT (bad → good)

Bad:

SELECT * FROM logs WHERE level='ERROR' ORDER BY created_at DESC LIMIT 100;


Good:

CREATE INDEX idx_logs_level_created ON logs(level, created_at);
-- then the same SELECT will use index to avoid filesort

Excellent question 👍 You’re asking whether we can store a list/array of product IDs in the order_items table for a given order_id.

❌ Why Not Store as a List/Array?

In relational databases like MySQL, tables are designed to follow 1NF (First Normal Form).

1NF Rule: A column must hold atomic values (not lists/arrays/CSV).

Example of violating 1NF:

order_id | product_ids
---------|----------------
1        | 10,11,12       -- ❌ breaks normalization


Problems:

Hard to query → "find all orders containing product 11" requires string parsing (FIND_IN_SET), very slow.

No foreign keys → you lose referential integrity with products(id).

Updates are messy → if you remove product 11, you rewrite the whole string.

✅ Correct Design: One Row Per Product

That’s why order_items table has one row per product in an order:

order_id | product_id | quantity | price
---------|------------|----------|-------
1        | 10         | 2        | 50.00
1        | 11         | 1        | 150.00
1        | 12         | 3        | 20.00


This way:

Easy to join with products table.

Easy to calculate totals with SUM(quantity * price).

Referential integrity is preserved via FOREIGN KEY (product_id).

⚡ If You Really Want an Array

MySQL itself doesn’t have a native ARRAY type like PostgreSQL, but you have some alternatives:

👉 Works, but you lose performance and referential integrity.
👉 Good for logs, not core transactional data.

JOIN order_items oi ON o.id = oi.order_id

🔹 3. If You Need amount in Orders Table

Use triggers to update it automatically.

✅ Example: MySQL Trigger to Maintain orders.amount
DELIMITER //

CREATE TRIGGER update_order_total
AFTER INSERT ON order_items
FOR EACH ROW
BEGIN
   UPDATE orders
   SET amount = (
       SELECT SUM(quantity * price)
       FROM order_items
       WHERE order_id = NEW.order_id
   )
   WHERE id = NEW.order_id;
END //

DELIMITER ;


Now, whenever a new item is added to an order, orders.amount is updated automatically.

Transactional Systems (OLTP): Use order_items and calculate totals dynamically.

Reporting Systems (OLAP, dashboards, BI): Store orders.amount (denormalized for performance), often using ETL or triggers.
